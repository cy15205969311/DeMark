# 稿定设计提取逻辑说明

## 1. 文档目的

本文档用于完整说明当前仓库中“稿定设计”提取能力的实现方式、代码入口、核心判断规则、分页提取策略、测试覆盖以及后续扩展方向。

这份文档描述的是当前代码已经落地的逻辑，重点对应以下文件：

- `core/image_extractor.py`
- `crawlers/gaoding_crawler.py`
- `utils/url_parser.py`
- `utils/image_validator.py`
- `tests/test_gaoding.py`

如果后续稿定页面结构发生变化，或者我们继续增加新的接口/字段识别方式，优先同步更新本文档，避免“代码已经变了，但文档还停在旧实现”。

## 2. 功能目标

当前稿定设计提取逻辑的目标不是“抓到页面上的所有图片”，而是尽可能稳定地拿到真正的设计稿页面结果，并统一输出为和图怪兽、创客贴兼容的套图结构。

当前目标包括：

1. 正确识别稿定分享链接。
2. 自动清理分享文案后缀，例如 `#我分享了...`。
3. 尽量优先返回更接近原图、无水印的结果。
4. 当分享页本身已经暴露出多页预览信息时，直接组装成整套 `imageUrls/pages/pageCount`。
5. 仅在稿定提取器内部修改逻辑，不影响其他平台和通用核心流程。

## 3. 当前代码入口

### 3.1 平台接入入口

稿定平台是在 `core/image_extractor.py` 中接入统一调度的。

关键点：

- `ImageExtractor.__init__()` 中注册 `GaodingCrawler`
- `_extract_local()` 中识别 `platform == 'Gaoding'`
- `_extract_gaoding()` 中把请求转交给 `GaodingCrawler.extract_image()`

这意味着稿定逻辑在系统架构上是“平台专用爬虫”，不会直接改写图怪兽、创客贴、Canva 的提取路径。

### 3.2 URL 解析入口

稿定链接解析在 `utils/url_parser.py` 中完成，当前主要负责两件事：

1. 平台识别：把 `gaoding.com` / `gaoding.cn` 识别为 `Gaoding`
2. 分享链接清洗：去掉 `#我分享了...` 这类分享文案 fragment

这一点非常重要，因为很多稿定分享链接是：

```text
https://www.gaoding.com/dam/share/xxxx#我分享了「...」
```

如果不去掉 fragment：

- 第三方接口可能收到污染后的 URL
- 日志会变难看
- 后续调试时容易误判实际请求地址

## 4. 整体提取流程

当前稿定提取流程可以概括为：

1. URL 清洗与平台识别
2. 进入 `GaodingCrawler.extract_image()`
3. 优先执行静态提取
4. 静态提取失败后，再走动态抓取
5. 将结果统一整理为单图/套图兼容结构

对应到 `crawlers/gaoding_crawler.py`，入口函数是：

```python
async def extract_image(self, url: str, extracted_params: Optional[Dict] = None) -> Optional[Dict]:
```

当前主流程顺序如下：

1. `_classify_link(url)` 判断是分享页、模板页、编辑页还是列表页
2. `_create_session()` 创建带稿定请求头的 `aiohttp` 会话
3. `_static_scraping(session, url, link_type)` 尝试静态提取
4. 如果静态提取失败且不是 listing，再走 `_dynamic_scraping(url, link_type)`
5. 返回统一结果，或 `None`

## 5. 链接类型识别

函数：`GaodingCrawler._classify_link()`

当前会把稿定链接分为以下几类：

- `share`
- `template`
- `editor`
- `creation`
- `detail`
- `listing`

判断依据主要来自：

- URL path，例如 `/dam/share/`、`/templates/`、`/editor/design`
- query 参数，例如 `templateId`、`workId`、`shareId`、`id`

设计目的：

- `listing` 类页面不适合做单设计稿提取，静态失败后不会继续做动态提取
- `share` / `editor` / `template` 类页面会进入完整提取流程

## 6. 静态提取是当前稿定的主路径

函数：`GaodingCrawler._static_scraping()`

这是当前稿定提取最重要的函数，因为真实线上稿定分享页已经能在 HTML 中暴露出足够多的结构化信息，没必要一开始就依赖 Selenium。

当前静态提取的候选来源共有 5 类：

1. `meta_entries`
2. `dom_entries`
3. `json_entries`
4. `text_entries`
5. `preview_bundle_entries`

其中对“套图提取”最关键的是第 5 类。

### 6.1 预览包提取：当前稿定套图的主实现

函数：`GaodingCrawler._extract_preview_bundle_from_html()`

这是本次稿定套图支持的核心。

它做的事情是：

1. 对 HTML 做 `html.unescape()`
2. 从整页文本中查找 `source_preview_info.url`
3. 如果 `source_preview_info.url` 不存在，再回退到 `preview_info.url`
4. 再解析 `extends_previews` 数组中的所有 `url`
5. 把主图和扩展页按顺序拼成一个 bundle
6. 去重后返回 `[(url, hint), ...]`

### 6.2 为什么优先 `source_preview_info`

稿定分享页里常见这两个字段：

- `preview_info`
- `source_preview_info`

经验上：

- `preview_info` 更像分享页预览图
- `source_preview_info` 更像更接近原始设计结果的图

所以当前实现顺序是：

1. 优先拿 `source_preview_info.url`
2. 如果它为空，再用 `preview_info.url`
3. `extends_previews` 继续补后续页面

这样做可以让“第一页质量”尽量更好，但不会丢掉后续页。

### 6.3 为什么 `extends_previews` 能做套图

通过真实稿定分享页分析，当前分享页本身会暴露：

- 一张主预览图：`preview_info`
- 多张扩展页图：`extends_previews`

对于你这次的真实案例，分享页里就是：

- 第 1 页在 `preview_info.url`
- 第 2~4 页在 `extends_previews[*].url`

所以当前代码把这组结构视为“分享页直接给出的分页结果”，优先级高于泛化的 JSON 路径分组。

### 6.4 静态提取中的优先级顺序

当前 `_static_scraping()` 中的优先级是：

1. `preview_bundle_entries` 直接组套图
2. `_select_best_multi_page_group(json_entries)` 从 JSON 路径组多页
3. `meta_entries` 选单图
4. `meta + json + dom + text` 全量候选选单图

也就是说：

- 只要分享页里有 `preview_info + extends_previews`，就会优先返回套图
- 只有静态套图信号不足时，才回到单图评分逻辑

## 7. 预览包提取的辅助函数

### 7.1 `_extract_object_url(text, field_name)`

作用：

- 从形如 `"preview_info": {"url": "..."}` 的对象中提取 `url`
- 兼容 `url: null`

当前用于：

- `source_preview_info`
- `preview_info`

### 7.2 `_extract_array_object_urls(text, field_name)`

作用：

- 从形如 `"extends_previews": [{...}, {...}]` 的数组中提取每个对象的 `url`
- 对结果执行标准 URL 归一化与去重

当前用于：

- `extends_previews`

## 8. 通用 JSON / DOM / 文本候选提取

除了直接处理预览包，稿定爬虫还保留了一套通用候选提取能力，用于兜底。

### 8.1 `_extract_meta_candidates()`

提取：

- `og:image`
- `twitter:image`
- `image_src`

适合：

- 模板详情页
- 某些只暴露封面的公共页

### 8.2 `_extract_dom_candidates()`

提取：

- `img.src`
- `data-src`
- `data-original`
- `data-url`
- `srcset`
- `source`
- `video poster`

适合：

- 页面已经把部分资源渲染到 DOM

### 8.3 `_extract_json_candidates()`

做法：

1. 遍历所有 `<script>`
2. 找出内容本身像 JSON 的脚本
3. `json.loads()` 后递归遍历字符串值
4. 只保留能归一化成图片 URL 的字符串

适合：

- 页面通过内嵌状态树输出数据

### 8.4 `_extract_contextual_urls_from_text()`

作用：

- 通过正则在整段 HTML 文本中扫描图片链接
- 适合处理无法直接 JSON 解析但仍然包含图片 URL 的情况

注意：

这类提取精度最低，所以只作为兜底。

## 9. 稿定多页通用分组逻辑

函数：`GaodingCrawler._select_best_multi_page_group()`

这套逻辑在稿定中仍然保留，用于处理“并非显式 `preview_info + extends_previews`，但 JSON 路径已经能表现出分页关系”的场景。

基本思路是：

1. 对每个候选 `(image_url, hint)` 做标准化
2. 使用 `_candidate_group_key(hint)` 计算组键
3. 把相同组键的候选聚合
4. 对组做评分，优先保留更像 page/artboard/pages 的路径
5. 如果一组内至少有 2 张图，且组分足够高，则认为是套图候选

### 9.1 `_candidate_group_key()` 的作用

举例：

- `jsonData.work.pages[0].image`
- `jsonData.work.pages[1].image`

会被规整到同一个组键，例如：

```text
jsonData.work.pages[].image
```

再通过去掉尾部 `.url/.image/.preview/.source` 等字段，进一步稳定分组。

### 9.2 正负路径提示

正向提示：

- `page`
- `pages`
- `artboard`
- `slide`
- `frame`
- `canvas`
- `render`

负向提示：

- `banner`
- `recommend`
- `sidebar`
- `widget`
- `header`
- `footer`
- `material`

本质上，这套逻辑是在尽量偏向“设计稿分页”，而不是“页面装饰资源”。

## 10. 动态抓取路径

函数：`GaodingCrawler._dynamic_scraping()`

只有静态提取失败时才会进入动态抓取。

当前动态路径依赖通用浏览器服务 `core/browser_service.py`，但稿定自身的解释和筛选逻辑仍然写在 `GaodingCrawler` 里。

动态路径顺序如下：

1. `browser_service.extract_dynamic_content()`
2. `_analyze_dynamic_data()`
3. 如果还不够，再调用 `browser_service.extract_images_from_page()`
4. 对页面可见图片做稿定自身筛选

### 10.1 `_analyze_dynamic_data()` 的分层优先级

当前优先级：

1. `pageSnapshots` 逐页快照
2. `visibleCandidates`
3. 合并后的 `pageSpecificImages` / `imageUrls` / `resourceUrls` / `jsonData` / `windowData`
4. 从合并结果中再尝试多页分组
5. 最后才回到单图

也就是说，稿定即使进入动态抓取，也仍然优先尝试“逐页结果”，而不是直接从整页里挑一张最大图。

## 11. 单图候选评分逻辑

函数：`GaodingCrawler._score_candidate()`

这是当前稿定在“无法直接确定多页结果”时的主要排序逻辑。

评分目标是：

- 提高真正设计图的优先级
- 降低侧边栏、图标、推荐位、带水印预览图的优先级

### 11.1 域名权重

当前对以下域名做了正向加分：

- `gaoding-market.dancf.com`
- `gaoding-market-fat.dancf.com`
- `*.dancf.com`
- `*.gaoding.com`

### 11.2 字段权重

当前对以下 hint 字段加权较高：

- `source_preview_info`
- `original_preview_info`
- `export_url`
- `download_url`
- `original_url`
- `source_url`
- `raw_url`
- `full_url`

设计原因：

- 这些字段通常比普通 `preview` 更接近真正输出结果

### 11.3 关键词权重

正向关键词包括：

- `export`
- `download`
- `original`
- `source`
- `preview`
- `cover`
- `page`
- `artboard`
- `slide`
- `frame`

负向关键词包括：

- `watermark`
- `with_wm`
- `with-watermark`

另外：

- 普通 `preview_info` 会被适度降权
- `source_preview_info` 不会被这条降权误伤

## 12. URL 归一化与防脏数据处理

函数：`GaodingCrawler._normalize_candidate_url()`

这个函数是稿定提取稳定性的关键之一。

### 12.1 它解决了什么问题

稿定分享页有时会把 URL 嵌在 HTML 转义或 JSON 片段里，比如：

```text
https://...jpg?auth_key=xxx","width":1242,"height":1656
```

如果不做归一化：

- 会把整段 JSON 尾巴一起当成 URL
- 验证阶段可能误判
- 浏览器打开后会得到错误页面

### 12.2 当前归一化步骤

1. `html.unescape()`
2. 处理 `\/`、`\u0026`、`\u002F`、`\u003A`、`\u003D`
3. 用 `IMAGE_URL_PATTERN` 从原始文本中只截出真正的图片 URL
4. 支持 `//host/path` 自动补成 `https:`
5. 去掉 fragment
6. 对稿定图片再做 `_strip_gaoding_oss_process()`

### 12.3 为什么移除 `x-oss-process`

之前真实日志已经出现过这种情况：

- 提取到的稿定图片链接带 `x-oss-process=image/resize`
- 或者因为截断问题只剩半截 OSS 参数
- 打开后稿定 OSS 直接返回“参数格式错误”

所以当前逻辑对稿定 / `dancf.com` 图片链接统一移除 `x-oss-process`，只保留签名和原始对象地址。

这样更稳定，也更符合“优先拿原始设计结果”的目标。

## 13. 候选去重逻辑

函数：

- `_candidate_key()`
- `_dedupe_entries()`
- `_dedupe_urls()`

当前去重时会忽略：

- `x-oss-process`
- `format`

设计原因：

- 这些参数更像“同一张图的不同处理形式”
- 对套图识别来说，应该把它们视为同一资源，避免重复页

## 14. 图片有效性校验

稿定图片的有效性校验依赖 `utils/image_validator.py`。

当前对稿定相关域名会自动补：

- `Referer: https://www.gaoding.com/`
- `Origin: https://www.gaoding.com`

这是为了降低：

- 防盗链失败
- HEAD / GET 被拒
- 看起来是“链接失效”，但实际上只是请求头不对

## 15. 输出结构

稿定返回结果与其他平台统一，形如：

```json
{
  "imageUrl": "https://example.com/page-1.jpg",
  "imageUrls": [
    "https://example.com/page-1.jpg",
    "https://example.com/page-2.jpg",
    "https://example.com/page-3.jpg"
  ],
  "pages": [
    { "page": 1, "imageUrl": "https://example.com/page-1.jpg" },
    { "page": 2, "imageUrl": "https://example.com/page-2.jpg" },
    { "page": 3, "imageUrl": "https://example.com/page-3.jpg" }
  ],
  "pageCount": 3,
  "isMultiPage": true,
  "platform": "Gaoding",
  "source": "html-preview-bundle",
  "method": "static_html_preview_bundle",
  "linkType": "share"
}
```

说明：

- `imageUrl` 兼容旧逻辑，始终是首张图
- `imageUrls` 才是完整套图列表
- `pages` 是 UI 和后续下载最友好的结构
- `source` / `method` 用于排查当前命中的提取路径

## 17. 当前实现的边界与限制

虽然稿定套图已经能覆盖当前真实分享页，但仍然有一些边界需要明确：

### 17.1 `auth_key` 是时效签名

稿定图片链接通常带 `auth_key`，这意味着：

- 当下提取成功，不代表链接永远可用
- 过期后重新打开会失效
- 这是平台签名机制，不是本地代码 bug

### 17.2 当前套图主路径依赖分享页直接暴露预览包

当前最稳定的多页路径是：

- `preview_info`
- `source_preview_info`
- `extends_previews`

如果未来稿定隐藏这些字段，当前套图率会下降，需要补新的结构解析。

### 17.3 当前没有把 `design-json` 作为主生产路径

真实稿定分享页里通常还会暴露一个 `design-json` 地址，里面可能包含 `pages[0].elements[*]` 等设计结构。

这为后续增强留下了空间，但当前代码没有把它作为正式主路径，原因是：

- 分享页预览包已经足够稳定
- 直接使用分享页暴露的预览图，复杂度更低
- 可以避免把更多内部设计结构耦合进当前提取器

## 18. 后续可扩展方向

如果后续还要继续加强稿定套图提取，建议优先考虑以下方向：

1. 新增 `design-json` fallback
   - 当 `extends_previews` 缺失时，尝试从 `design-json` 结构推断页数和页面顺序
2. 更细的多页排序
   - 对 `preview_info` 与 `extends_previews` 做尺寸/时间/字段来源二次校验
3. 更强的动态页识别
   - 对编辑页中底部页签、左侧缩略图和分页切换行为做稿定专用解析
4. 更强的原图能力
   - 如果稿定后续暴露更稳定的 `export_url` / `download_url` 套图字段，可直接取代当前预览图套图路径

## 19. 维护建议

维护稿定提取器时，建议遵循以下原则：

1. 优先在 `crawlers/gaoding_crawler.py` 内解决问题。
2. 只有当通用浏览器服务确实无法满足时，才考虑动 `core/browser_service.py`。
3. 每修一次真实链接问题，都至少补一条回归测试。
4. 如果日志里出现新字段，优先先在 HTML 中确认它是否稳定，再决定是否接入正式逻辑。
5. `source` 和 `method` 命名尽量保留可读性，方便日志排查。

## 20. 结论

当前稿定设计提取逻辑的核心原则可以概括为一句话：

优先吃分享页已经公开暴露的结构化预览信息，把真正的设计稿分页结果直接组装出来；只有当静态结构不足时，才逐步回退到通用 JSON 分组、动态快照和单图评分。

这样做的好处是：

- 改动范围局限在稿定提取器内部
- 不会影响图怪兽、创客贴、Canva 的已有逻辑
- 对真实稿定分享页的命中率高
- 排障路径清晰
- 测试可维护性较好
