# 818ps 套图提取逻辑说明

本文档专门说明 DeMark 当前针对 818ps / 图怪兽分享链接的“整套设计稿提取”实现逻辑，便于后续维护、排错和继续优化。

## 目标

输入一条 818ps / 图怪兽分享链接，尽可能稳定地拿到：

- 正确的整套设计稿图片
- 正确的页数与页序
- 不包含编辑器素材、小图标、文字素材、水印预览图的结果

最终输出统一为：

```json
{
  "imageUrl": "首张设计稿",
  "imageUrls": ["第1页", "第2页", "第3页"],
  "pages": [
    { "page": 1, "imageUrl": "第1页" },
    { "page": 2, "imageUrl": "第2页" },
    { "page": 3, "imageUrl": "第3页" }
  ],
  "pageCount": 3,
  "isMultiPage": true
}
```

## 整体链路

### 1. 总调度入口

入口在 [core/image_extractor.py](./core/image_extractor.py) 的 `ImageExtractor.extract_image()`。

它会按下面的顺序执行：

1. 解析分享链接，识别平台和参数
2. 尝试第三方 API 网关
3. 调度到 818ps 本地提取器
4. 必要时进入 Selenium / Chrome 动态抓取兜底
5. 调用 `_normalize_result_images()` 统一结果结构

`_normalize_result_images()` 的作用很关键，它会把单图和多图统一补齐成：

- `imageUrl`
- `imageUrls`
- `pages`
- `pageCount`
- `isMultiPage`

这样 GUI、下载器和后续导出逻辑就不需要分别处理两套结构。

## 818ps 提取器的核心策略

主逻辑在 [crawlers/tuguaishou_818ps.py](./crawlers/tuguaishou_818ps.py) 的 `Tuguaishou818psCrawler.extract_image()`。

### 2. 分享参数提取

`_extract_share_query_params()` 会从分享页 URL 里解析常用参数：

- `picId`
- `upicId`
- `share_id`
- `share_uid`
- `save_type`
- `user_source`

这些参数决定后续是否可以直接调用官方分享 API。

### 3. 官方分享 API 优先

如果 URL 中拿到了 `share_id`，会优先调用 `_extract_from_share_api()`。

这里会尝试几个官方接口：

- `apiV1/template/index/share-template`
- `api/team-share-get-templ`
- `apiv2/get-template-page-data`
- `apiv2/get-template-info`

这样做的原因很直接：

- 分享 API 返回的数据最接近“真实页面定义”
- 如果能直接拿到整套预览图，就没必要把成功率全部压在浏览器模拟上
- 官方数据里通常能直接推断总页数，便于校验是否漏页

### 4. 从官方 API 中提取整套预览

`_extract_share_api_preview_groups()` 会从多个 API 返回体中收集可用预览图组，并推断期望页数。

它会优先收集这些字段：

- `share_template.data.preview`
- `team_share_get_templ.preview`
- `team_share_get_templ.page_map`
- `team_share_get_templ.doc.pageAttr.pageInfo`
- `get_template_page_data.url.preview`

同时会调用 `_extract_expected_page_count_from_share_api_payloads()` 计算“理论应有页数”。

如果某一组 URL：

- 经过规范化后仍然有效
- 通过了设计稿候选过滤
- 通过了图片可访问验证
- 数量达到理论页数，或至少大于 1

就会直接构造成整套结果返回。

## 为什么要过滤掉某些 URL

### 5. `user_preview_ue` 优先

当前逻辑会优先保留类似下面这类地址：

```text
https://img.tuguaishou.com/user_preview_ue/...
https://img.tuguaishou.com/user_preview/...
```

原因：

- 这类路径更接近最终设计稿预览
- 在多页分享场景里稳定性更高
- 一般尺寸更大、清晰度更高

### 6. 排除 `ips_user_preview_api`

当前逻辑明确把 `ips_user_preview_api` 视为低优先级甚至排除项。

原因：

- 这类结果经常是带水印或偏业务态的预览
- 有时会和真正的设计稿页混在一起
- 容易导致“第一页原稿 + 第一页带水印图”同时被识别出来，造成误判

### 7. 其他排除项

`_is_design_page_candidate()` 和浏览器侧 `_score_preview_candidate()` 都会对这类资源做强过滤：

- `ips_svg/`
- `ips_group_word/`
- `ips_icon/`
- `ips_material/`
- `editor/`
- `material/`
- `element/`
- `asset/`
- `watermark`
- `toolbar`
- `button`
- `crown`
- `badge`
- `vip`
- `icon`
- `logo`

这些大多是编辑器素材、角标、按钮、会员标识或文字资源，不应该混入整套设计稿结果。

## 浏览器动态抓取回退

当官方分享 API 没法直接给出完整结果时，会进入浏览器动态抓取。

核心实现在 [core/browser_service.py](./core/browser_service.py) 的 `extract_dynamic_content()`。

### 8. 动态抓取分两步

第一步：整页滚动采样

- 先打开动态页
- 抓一次页面快照
- 连续做多轮滚动采样
- 每轮记录：
  - 当前识别到的图片 URL
  - 页面标记，如“第1页”“第2页”
  - 资源 URL
  - 当前焦点页

第二步：逐页激活

- 根据滚动阶段识别出的页面标记拿到页码
- 逐页尝试激活对应页面
- 每次激活后再抓一轮页面快照
- 从该页的快照里选出最像“真正设计稿预览”的 URL

这样做的目的，是把“页面里所有资源”与“某一页真正对应的主预览图”区分开。

### 9. 浏览器为什么既滚动又逐页激活

只滚动不够，原因有两个：

- 有些页面资源只有在对应页真正进入焦点后才会出现
- 仅靠全局资源池，容易把字体图、素材图、按钮图一起混进去

所以现在的做法不是“只靠滚动找图”，而是：

1. 先滚动，找页码与初始资源池
2. 再逐页激活，提取每一页专属的预览候选

这也是当前能够稳定提取整套设计稿的关键。

## 动态页候选图是如何选出来的

### 10. 候选收集

`_collect_dynamic_image_candidates()` 会统一收集这些来源：

- 浏览器直接看到的图片 URL
- 浏览器资源加载列表
- 逐页激活得到的 `pageSpecificImages`
- 每一页快照里的 `previewUrls` / `newUrls`
- 页面源码中的 URL
- JSON 数据块里的 URL
- `window` 对象里的 URL

### 11. 逐页专属结果提取

`_extract_page_specific_image_urls()` 会按页遍历 `pageSnapshots`，对每一页：

1. 收集该页快照中的候选 URL
2. 先做设计稿过滤
3. 再按评分排序
4. 调用 `_validate_image_entry_urls()` 验证图片是否真实可访问
5. 每页选出 1 张最合适的图

这样最终会形成按页排序的 `page_specific_urls`。

### 12. 动态候选分析与合并

`_analyze_dynamic_data()` 是动态页结果归并的核心：

1. 先看是否识别到了页面标记 `pageMarkers`
2. 再提取逐页专属结果 `page_specific_urls`
3. 再从全量动态数据里收集候选图
4. 对候选图做设计稿过滤
5. 对过滤后的结果做图片可访问校验
6. 把 `page_specific_urls` 和全局有效设计稿图合并去重
7. 如果识别到的页数多于当前有效图片数，打印警告日志

这一层的目标是：

- 优先信任逐页激活得到的结果
- 但又不浪费全局抓取里补充出来的有效设计稿图

## 结果构造与归一化

### 13. 多页结果构造

`_build_multi_image_result()` 会把最终结果整理成统一格式：

- 第一张放到 `imageUrl`
- 全量放到 `imageUrls`
- 自动生成 `pages`
- 自动填充 `pageCount`
- 自动计算 `isMultiPage`

### 14. 主提取器再做一次结构归一化

即使 818ps 提取器已经返回了多页结果，`ImageExtractor._normalize_result_images()` 仍然会再次兜底归一化，保证：

- 单页结果也具备 `pages`
- 老逻辑只读 `imageUrl` 也不会挂
- GUI 和下载器始终能读取 `imageUrls`

## 下载阶段

下载逻辑在 [utils/downloader.py](./utils/downloader.py)。

### 15. 批量下载行为

`batch_download()` 会：

- 接收整套图片 URL 列表
- 按并发数限制批量下载
- 自动生成文件名
- 返回成功数、失败数、每张图的下载结果

GUI 层在 [gui/main_window.py](./gui/main_window.py) 中会根据 `pageCount` 判断：

- 单页：走单图下载
- 多页：直接走整套批量下载

## 排错建议

### 16. 只抓到第一页时看什么

优先看日志里是否出现：

- `页面标记`
- `动态页滚动采样`
- `动态页逐页激活`
- `pageSpecificImages`

如果有页面标记，但最终有效图数量偏少，优先排查：

- 某些页没有激活成功
- 逐页快照中只出现了素材图，没有出现真正预览图
- 图片校验把某些候选图过滤掉了

### 17. 抓到了“原图 + 带水印图”时看什么

优先检查过滤规则是否放进了：

- `ips_user_preview_api`
- `ips_svg`
- `ips_group_word`

这通常不是整套设计稿页面本身。

### 18. 页数不对时看什么

优先对比三个数量：

- `pageMarkers` 识别出的页数
- 官方分享 API 推断出的 `expected_page_count`
- 最终 `imageUrls` 的数量

如果前两者大于第三者，说明当前结果仍然存在漏页，可以继续从：

- 逐页激活方式
- 快照评分规则
- 图片验证阈值

这三处继续优化。

## 一句话总结

当前这套实现的核心不是“盲目抓页面上所有图片”，而是“先拿官方分享数据，再用浏览器逐页激活补齐页面专属预览，最后通过设计稿过滤和图片校验把整套结果干净地组出来”。这也是现在能稳定提取 818ps 套图的关键。
